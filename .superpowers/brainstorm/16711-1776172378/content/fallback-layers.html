Fallback 不是二元的——是 3 层降级

你说"OCR 是 fallback",但 fallback 不应该被触发得太频繁,否则主路径失去意义。我建议分 3 层:card → card with inline fix → full form。大多数小问题应该在第 2 层解决,永远不掉到第 3 层。

Layer 3 · Full form(你现在的 submit.html)

触发条件:OCR 完全失败(图太糊/不是发票) · 多单据 PDF · 查重撞单需解释 · 用户主动点"修改" · 没有成本中心/员工信息缺失

原则:这层永远存在作为 safety net,但要让用户"感觉不到它的存在"——只有少数情况会来到这
目标占比:<5%
两个设计决策你要拍板:

① Layer 2 的"inline fix"范围划到哪?
我的主张:最多 2 个字段需要 inline fix 还留在 Layer 2。如果 ≥3 个字段都有问题,直接降级到 Layer 3。否则 card 会变成一个残缺的 form,违背设计原则。

② 下面这些 edge case 你觉得应该在哪层?
• 发票查重撞单 → Layer 2 card 里加一句"检测到 X 月 Y 日已报销,是误操作吗?[是/否]",还是 Layer 3?
• 预算 blocked(超阈值) → Layer 2 card 里给 2 个选项(换成本中心 / 走特批),还是 Layer 3?
• 多页 PDF 检测到 >1 张票 → Layer 3 没法处理,需要新设计(splitter UI)
• 分类置信 < 50% → Layer 2 的 inline chip,让用户选?
• 同一商户历史报销 3 次都是"交通",这次识别成"餐饮" → Layer 1 card 里加一句"上次在这报的是交通,是同一类消费吗?"
