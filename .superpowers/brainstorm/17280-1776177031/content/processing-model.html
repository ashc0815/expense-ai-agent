决策 ③ 用户上传后,等 card 出现的时候看到什么?

从按下"上传"到 card 准备好,中间要跑 OCR + 分类 + 查重 + 预算检查 4 个 tool,体感 2-5 秒。这几秒用户看到的东西决定了"感觉是 agent 在工作"还是"感觉卡住了"。

正在处理...
通常 2-5 秒

A · 阻塞 loader(一个转圈)

按下上传后,整个页面变成"处理中"状态,成功后一次性显示 card。

好处:最简单,一屏只有一个状态,没歧义
代价:2-5 秒的"没反应"体感会让人怀疑是不是卡了
🔍 识别发票...
¥358 · 海底捞
⏳ 分类中...
⏳ 查重中...

B · 渐进式填充 card

card 立刻出现,但字段一个一个"点亮"——金额先出(OCR 最快)→ 分类填上 → 预算徽章亮起。完成前按钮灰着。

好处:每 500ms 都有视觉反馈,感觉 agent 在"当面做事";技术上和 tool_call SSE 事件天然对齐
代价:实现复杂,要和 agent 的 streaming 事件耦合;card 在动会让一部分用户想"等它稳了再点"
✓ 已收到,正在后台处理
处理中...
可以再上传下一张 👇
📄

C · 乐观 + 队列(Ramp 本尊)

上传完立刻返回"已收到",进到一个"待处理/已就绪"的列表页。card ready 后 badge 变绿,用户再点进去提交。同时可以继续上传下一张。

好处:彻底去除"等"的感觉,批量报销场景无敌(出差回来 10 张票)
代价:页面结构完全重设计,不再是单张流程;需要一个"草稿列表页"承载队列;单张场景会被迫多一步(上传→列表→点卡)
我的倾向:B(渐进式填充)
理由:
• 你现在的 agent 本来就是 SSE streaming 的(assistant_text、tool_call、tool_result、draft_updated),直接把这些事件映射到 card 的字段上,技术上顺水推舟
• 2-5 秒里让用户看到"¥358 出来了 → 分类在跑 → 预算检查完成",体感就是"AI 正在帮我做 due diligence",这是整个产品叙事的核心
• C 是 Ramp 本身的做法,但它对应的是"员工一次报 20 张票 + 有 inbox 心智"的企业;你的场景大部分是"单张即刻报",列表页反而多一步

需要确认一件事:如果选 B,提交按钮在所有 tool 还没跑完时应该是灰的(不能点),还是始终亮着(用户可以在 AI 还在检查时就抢先提交)?我的默认是灰着——防止用户跳过查重/预算检查。
