A · Agent 自己降级

agent 检测到硬触发(OCR 完全失败、多页 PDF、≥3 个字段需改)就自动 redirect到 submit.html,带上已识别的部分预填。

好处:用户不用思考,能进 card 就进 card
风险:页面"突然跳走"会吓到用户,失去控制感

B · Agent 建议,用户点选

agent 检测到问题后,留在 quick.html 里用一个醒目的提示条,给 2 个选项:[手动填表] [重拍]。用户点「手动填表」才跳到 submit.html。

好处:用户保留控制,不会被偷袭;也能顺便引导 retry(重拍经常是更好的方案)
风险:多了 1 次 tap,有些用户会怕"重拍"而直接选手填

C · 混合:硬错自动,软错询问

分两种情况:硬错(OCR 完全 null、不是票)= agent 自动降级,反正没别的选项;软错(图糊、多字段错)= 留在 card 里让用户选。

好处:没得选的时候不浪费用户 1 次 tap;有得救的时候保留控制
代价:两条逻辑要维护,"硬"和"软"的边界要定清楚
我的倾向:C(混合)
理由:A 会让用户失去控制感(页面突然跳走最容易让人骂街),B 在"完全没救"的情况下还让用户点一下是浪费。混合能两全。
硬/软边界建议:
• 硬:OCR 返回空 / 文件不是发票(图片无金额+无商户)/ 多页 PDF 检测到 >1 张票
• 软:图糊但还能认出金额和商户(≥2 字段高置信)/ 分类全失败但字段齐全 / ≥3 字段需用户改
2b 附带问题——回流怎么做?
用户掉到 submit.html 之后,如果他填完发现"诶其实这张票没那么糟",能不能回到 card?

我的建议:不做回流。Layer 3 是终局——进去之后走传统提交路径,提交成功直接回列表页。理由是:一旦进 form,用户已经切换到"我要填字段"的心智模式,再把他拉回 card 反而增加困惑;并且 Layer 3 的触发率 <5%,不值得为它做双向状态机。

但如果你觉得"用户从 quick 跳来,应该能返回",我们可以加一个 submit.html 顶部的返回按钮("← 返回 quick")——纯 UI 层面,不是状态回流。
