入口和 Layer 之间怎么流转？

card/inline/form 的定义定下来了,现在要定义用户怎么进入以及layer 之间怎么切换。下面是 3 个入口设计决策,请一个一个看。

决策 ①

入口 URL — 是不是还停在 /employee/submit.html？

/employee/submit.html
📄 上传发票
↓ 上传后 3 步 wizard 被 card 替换
✓ 海底捞 ¥358 · 提交

A · 同一个 URL,内容形态替换

URL 不变。上传前是 dropzone,上传后 wizard 消失,直接替换成 card。

好处:入口不变,书签/快捷方式不破;灰度发布容易
代价:submit.html 逻辑更重,原来的 3 步 wizard 要和 card 共存一段时间
/employee/quick.html 新
📄 拍照 / 上传
✓ card
/employee/submit.html ← 手动填表 fallback

B · 新 URL,旧表单沦为 fallback 页

新建 /employee/quick.html 作为主入口。submit.html 只保留当 fallback 表单用。

好处:代码干净,两条路径互不污染,Ramp agent 流是"一等公民"
代价:需要改菜单/首页按钮,短期内"上传发票"点哪个要想清楚
我的倾向:B(新 URL)
理由:你现在的 submit.html 是 716 行,里面有 3 步 wizard、OCR 置信渲染、chat drawer、budget signal、post-submit timeline——再往里塞 card 会让它爆炸。新建 quick.html 把 agent 流写得干净,submit.html 只做 fallback,两个文件各自只做一件事。

但是你可能更在意"老用户不用换链接",所以这是个产品 vs 工程的权衡,想听你的判断。
